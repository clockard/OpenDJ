Overview

Among other things, the OpenDS project quality process involves :

  • Thorough code reviews by peer and fellow community members
  • Maximizing code coverage through unit tests and carefully wrapped integration tests
  • Continuous build and integration with Hudson
  • Advanced user oriented test with Staf/Stax
  • Robustness testing with tools like OULU
  • Protocol compliance testing with independent third-party tools like the OpenGroup VSLDAP suite

The quality in the OpenDS project is not the result of a distinct team of "Quality Assurance" experts but it is the responsibilty of each individual contributor.

Contributing Tests

If you want to participate in the project by adding tests or fixing broken or deprecated tests, then you will first need to complete and submit the Sun Contributor Agreement as outlined in the OpenDS Governance Model. Once this agreement has been accepted, you can forward the confirmation to contribution-request@opends.dev.java.net to request the Contributor role. An OpenDS committer will be assigned to work with you. Individuals who have been given the Contributor role will not initially be allowed to commit changes directly into the repository, but active Contributors may be granted the Committer role over time.

Unit Tests

Unit testing consist of testing the basic units of a given piece of software. Each test must be a single and as specific as possible question. There are many different types of unit tests:

  1. tests that define what must work
  2. tests that define what must not happen
  3. tests that define consistence between symmetrical (or reciprocal) methods

The unit tests are available in the OpenDS repository in {/path/to/workspace}/opends//tests/unit-tests-testng. The repository contains all the libraries you need to build and run the tests. To run the unit tests, use the test target in the main build.xml file.

To do so, change to {/path/to/workspace}/opends and :

  • on Unix, run: ./build.sh test
  • on Windows, run: build.bat test

The build also comes with a very useful tool to drive your test writing: code coverage. Once you've written your set of tests and run them successfully, you may want to know how good those tests actually are in covering your code. Just use the testwithcoverage target.

To do so, change to {/path/to/workspace}/opends and :

  • on Unix, run: ./build.sh testwithcoverage
  • on Windows, run: build.bat testwithcoverage

At the end of the build, the code coverage tool will provide you with an html report that you will find in {/path/to/workspace}/opends/build/coverage/report/index.html. Use it to figure out what methods you've missed in your testing and improve your test suite.

Integration Tests

Integration testing aims at checking that the interfaces to the outside world allow easy and smooth integration of the test object (i.e. the product) in an existing system.

You can find this suite in {/path/to/workspace}/opends/tests/integration-tests-testng.

On Unix and Linux, simply change to {/path/to/workspace}/opends/tests/integration-tests-testng and run build.sh. This will build the tests and generate another script test.sh that you will have to execute to actually run the tests.

We are in the process of moving away from the shell scripts and porting everything to ant to be as platform independent as possible.

Functional Tests

System functional testing is a form of software testing that attempts to determine whether each function of the system works as specified.

This is not available in the repository as of today but we are working on a prototype that would let us test the features in more complex deployment schemes than the other suites currently permit.
