1. Introduction

1.1 Purpose

This document serves as a feature requirement specification for Schema Synchronization. 

The intended audience for this document includes anyone who would like to understand the functionality of the product and would like to evaluate its features to suit their needs.

1.2 Terminology

The priorities of requirements that will be used in this document include:

  • Must Have (MH) -- These requirements are those which are absolutely required for the implementation of this feature. An implementation that do not satisfy those requirements is not acceptable.

  • Highly Desirable (HD) -- These requirements are those which are very important to include in the implementation of this feature but are not so important as to be considered absolute requirements. It is likely that they will be included, but if necessary is is acceptable to have an implementation that do not satisfy those requirements.

  • Nice to Have (NTH) -- These requirements are those which may provide some level of benefit but are not critical to the implementation of this feature. They may be included in the implementation of this feature if time permits, but priority must be given to the "must have" and "highly desirable" requirements.


The words method and means identify some kind of technique by which the feature will be supported. They do not specifically imply either an API (programming language method or procedure) or a command.

2. General Description

2.1 Feature Perspective


Schema management is a core feature of the Directory Service. Schema describe the entries that can be stored in the Directory Servers.

Synchronization is also a core feature of a Directory Service, essential to a scalable and highly available service.

It is therefore important that the changes made to the schema of an individual Directory Server are synchronized on all the Directory Servers contributing to the same Directory Service.

Schema synchronization must currently be done when the schema is modified in any of three possible ways:

  • By modifying the cn=schema suffix when the server is online.
  • By using the dedicated task to perform dynamic schema updates via file when the server is online.
  • By modifying the underlying backend files directly when the server is offline.
In some rare cases it is useful to have different schema on different Directory Servers even when the servers share all or part of their data. In those cases it must be possible to disable the schema synchronization or to specify a restricted list of servers that participate in schema synchronization.
The most common case is to share schema, however. Schema synchronization should therefore be enabled by default whenever synchronization is enabled.

2.2 User Characteristics

Administrators are going to use this feature to control schema synchronization.

2.3 Constraints

Schema modifications happen only when deploying new applications or new types of data. The rate of change for schema is therefore expected to be low. The schema synchronization implementation should therefore favor simplicity over high scalability.

2.5 Future Requirements

OpenDS will probably allow Multiple Schemas in the future. In this case different schema will be associated with different portions of the LDAP tree. Schema Synchronization may need to be extended when support for multiple schemas arrives. It is also possible, though, that support for multiple schemas will be provided by adding extra information to the schema to specify which part of the schema applies to each specific part of the tree. In this case schema synchronization may simply involve synchronizing the entire schema everywhere. If that happens, schema synchronization will continue to work without modification.

 

3. Specific Requirements

3.1 Administration Requirements

(MH) Schema Synchronization must be configurable though a CLI, or through the GUI.

3.2 Performance Requirements

Schema modifications occur at a very low rate. As a result, there are no performance requirements for Schema Synchronization.

3.3 Scalability Requirements

Schema modifications occur at a very low rate. Therefore schema synchronization need not be highly scalable in terms of number of modifications it can handle.
(MH) There must not be any limit to the number of servers whose schema can be synchronized.

3.4 Internationalization Requirements

N/A

3.5 Help Requirements

N/A

3.6 Security Requirements

(MH) Non authenticated servers must not be able to push or pull schema changes.
(HD) It should be possible to encrypt schema changes.

3.7 Tests and Maintenance Requirements

(MH) All synchronization tools must also work for schema synchronization including monitoring, full synchronization, entry diff, and so forth.

3.8 Conformance to Standards Requirements

They are no specific requirements to follow an existing standard.

3.9 Integration with other subsystems Requirements

(MH) This feature should have no impact on any other feature of OpenDS.
(MH) Schema Synchronization must be independent of the format of information stored in the schema. Should the storage format change, or new types of information such as ACIs be stored in the schema, Schema Synchronization must continue to work unchanged.
