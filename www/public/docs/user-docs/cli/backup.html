The backup Utility

The backup utility may be used to archive the contents of one or more Directory Server backends.

In order to use this tool, the --backupDirectory option must be used to specify the location to which the backup files should be written. It is also necessary to provide either one or more instances of the --backendID argument or the --backUpAll argument to specify which backend(s) to include in the archive. The complete set of arguments that may be used is as follows:
  • -d {path}
    --backupDirectory {path}

    This argument specifies the path to a directory into which the backup files should be placed. If multiple backends are to be archived, then a subdirectory will be created below this path for each backend, otherwise the backup files will be placed directly in this directory. Note that multiple backups for the same backend may be placed in the same directory, and if an incremental backup is to be performed, then the backup directory must already contain at least one full backup. This is a required argument.

  • -n {backendID}
    --backendID {backendID}

    This argument specifies the backend ID of the backend to be archived. This argument may be provided multiple times to indicate that multiple backends should be archived. The available backends in the server may be determined using the list-backends tool.

  • -a
    --backUpAll

    This argument indicates that the server should back up all configured backends. This option must no be used in conjunction with the --backendID argument.

  • -I {backupID}
    --backupID {backupID}

    This argument may be used to specify an identifier to use for the backup. If this is not provided, then a backup ID will be generated based on the current time. The backup ID must be unique among all backups in the provided backup directory.

  • -i
    --incremental

    This argument indicates that the server should perform an incremental backup rather than a full backup. An incremental backup should include only that data that has changed since a previous incremental or full backup, and therefore taking an incremental backup may be notably faster than a full backup. When restoring an incremental backup, it is first necessary to restore the original full backup and then any intermediate incremental backups, which may make the restore process somewhat slower than restoring just a full backup.

    Note that some types of backends may not support performing incremental backups, and in that case this option will be ignored and a full backup will be performed.

  • -B {backupID}
    --incrementalBaseID {backupID}

    This argument may be used to specify the backup ID for the existing backup against which to take an incremental backup. If this is not provided, then the incremental backup will be based on the latest incremental or full backup contained in the backup directory.

  • -c
    --compress

    This argument indicates that the server should compress the contents of the backup archive. The compression algorithm used may vary based on the backend type.

  • -y
    --encrypt

    This argument indicates that the server should encrypt the contents of the backup archive.

  • -h
    --hash

    This argument indicates that the server should generate a message digest of the contents of the backup archive, which may be used as a checksum during the restore process to ensure that the backup has not been altered.

  • -s
    --signHash

    This argument indicates that the server should cryptographically sign the hash generated with the --hash argument, which provides even stronger assurance that neither the backup archive nor the hash of its contents have been altered.

  • -H 
    -?
    --help

    This argument will display usage information for the backup utility and will exit without attempting to perform a backup.
