The ldappasswordmodify Utility

The ldappasswordmodify utility may be used to change or reset user passwords with the LDAP password modify extended operation as defined in RFC 3062. Using this mechanism for changing user passwords offers a number of benefits over a simple LDAP modify operation targeted at the password attribute, including:
  • It will allow users to change their own passwords even after they have expired, provided that this capability is allowed in that user's password policy.
  • It provides a mechanism for supplying the clear-text version of the current password, for further validation of the user's identity.
  • The user for whom the password is to be changed may be given in the form of an authorization ID rather than requiring the full DN.
  • If a new password is not provided, then the server may generate one for the user, provided that this capability is allowed in that user's password policy.

The complete set of arguments available for use with this tool includes:
  • -h {address}
    --hostname {address}

    This argument specifies the address to use to communicate with the Directory Server. It may be an IP address or a resolvable name. If this is not provided, then a default value of "localhost" will be used.

  • -p {port}
    --port {port}

    This argument specifies the port to use to communicate with the Directory Server. It should be an integer value between 1 and 65535, inclusive. If this is not provided, then a default port of 389 will be used.

  • -Z
    --useSSL

    This argument indicates that the communication with the Directory Server should be protected using SSL. If this option is provided, then the --port option should be used to specify the secure port for the server. This argument must not be used in conjunction with the --useStartTLS option. For more information about using SSL, see this page.

  • -q
    --useStartTLS

    This argument indicates that the communication with the Directory Server should be protected using the StartTLS extended operation. This argument must not be used in conjunction with the --useSSL option. For more information about using StartTLS, see this page.

  • -D {dn}
    --bindDN {dn}

    This argument specifies the DN to use when binding to the Directory Server through simple authentication. If this argument is not provided, then the --authzID option must be given to provide the authorization ID for the target user, and either the --currentPassword or --currentPasswordFile argument must be given to provide the current password for the user (this mode of use will be required for users to change their passwords after they have expired).

  • -w {password}
    --bindPassword {password}

    This argument specifies the password to use when authenticating to the Directory Server. This option must not be used in conjunction with the --bindPasswordFile argument.

  • -j {path}
    --bindPasswordFile {path}

    This argument specifies the path to a file containing the password to use when authenticating to the Directory Server. This option must not be used in conjunction with the --bindPassword argument.

  • -a {authzID}
    --authzID {authzID}

    This argument specifies the authorization ID that should be used to indicate whose password is to be changed. The authorization ID may be in the form "dn:" followed by the DN of the target user, or "u:" followed by the username of the target user. If this option is not given, then no authorization ID will be included in the request and the password for the authenticated user will be changed. This argument must not be used in conjunction with the --provideDNForAuthzID option.

  • -A
    --provideDNForAuthzID

    This argument indicates that the value specified by the --bindDN argument should be explicitly provided to the server as the authorization ID (i.e., the authorization ID provided will be "dn:" followed by the DN provided using the --bindDN option). This argument must not be used in conjunction with the --authzID option.

  • -n {password}
    --newPassword {password}

    This argument specifies the new password that should be assigned to the target user. This must not be used in conjunction with the --newPasswordFile option. If neither of these arguments is given, then the server will automatically generate a new password for the user, provided that a password generator is configured in the user's password policy.

  • -N {path}
    --newPasswordFile {path}

    This argument specifies the path to a file containing the new password that should be assigned to the target user. This must not be used in conjunction with the --newPassword option. If neither of these arguments is given, then the server will automatically generate a new password for the user, provided that a password generator is configured in the user's password policy.

  • -c {password}
    --currentPassword {password}

    This argument specifies the current password for the user. It must not be used in conjunction with the --currentPasswordFile argument. The user's current password must be provided for cases in which no authentication is performed (e.g., if a user is trying to change his/her password after it has already expired), and may also be required by the server based on the password policy configuration even if a bind password was given.

  • -C {path}
    --currentPasswordFile {path}

    This argument specifies the path to a file containing the current password for the user. It must not be used in conjunction with the --currentPassword argument. The user's current password must be provided for cases in which no authentication is performed (e.g., if a user is trying to change his/her password after it has already expired), and may also be required by the server based on the password policy configuration even if a bind password was given.

  • -X
    --trustAll

    This argument indicates that the client should trust any SSL certificate that the server may present when performing secure communication with either SSL or the StartTLS extended operation. This is convenient for testing purposes, but it is generally not recommended for use in a production environment because it could make the client susceptible to a man-in-the-middle attack. If this option is not provided, then the --trustStorePath option should be used to specify the path to a trust store to consult in order to determine whether to trust a server certificate.

  • -K {path}
    --keyStorePath {option}

    This argument specifies the path to a client key store file that should be used for secure communication via SSL or the StartTLS extended operation. This option is generally only required if the client needs to present its own certificate to the server, such as when performing SASL external authentication.

  • -W {password}
    --keyStorePassword {password}

    This argument specifies the password needed to access the contents of the client key store. This is only required if the --keyStorePath option is used. This option must not be used in conjunction with the --keyStorePasswordFile argument.

  • -u {path}
    --keyStorePasswordFile {path}

    This argument specifies the path to a file containing the password needed to access the contents of the client key store. This is only required if the --keyStorePath option is used. This option must not be used in conjunction with the --keyStorePassword argument.

  • -P {path}
    --trustStorePath {option}

    This argument specifies the path to a clien trust store file that should be used for secure communication via SSL or the StartTLS extended operation. This argument is not needed if the --trustAll option is not used, although it is recommended that a trust store be used when working in a production environment.

  • --trustStorePassword {password}

    This argument specifies the password needed to access the contents of the client trust store. This is only required if the --trustStorePath option is used and the specified trust store requires a password in order to access its contents (most trust stores do not require this). This option must not be used in conjunction with the --trustStorePasswordFile argument.

  • -U {path}
    --trustStorePasswordFile {path}

    This argument specifies the path to a file containing the password needed to access the contents of the client trust store. This is only required if the --trustStorePath option is used and the specified trust store requires a password in order to access its contents (most trust stores do not require this). This option must not be used in conjunction with the --trustStorePassword argument.

  • -H
    --help

    This argument will display usage information for the ldappasswordmodify utility and will exit without attempting to perform any additional processing.
