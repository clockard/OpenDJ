1. Introduction

1.1 Purpose

This document serves as a feature requirement specification for the OpenDS Client Tools.  Such tools will facilitate configuration, management, and monitoring of both individual servers and the directory service as a whole as well as provide facilities to manage and query the data within the service itself.   The scope of this document does not include include an installer program although it is conceivable that some level of administration could be performed via these tools.

The intended audience for this document includes all who would like to understand the functionality of the product and would like to evaluate its features to suit their needs.

1.2 Terminology

The priorities of requirements that will be used in this document include:

  • Must Have (MH) -- These requirements are those which are absolutely required for the implementation of this feature. An implementation that do not satisfy those requirements is not acceptable.

  • Highly Desirable (HD) -- These requirements are those which are very important to include in the implementation of this feature but are not so important as to be considered absolute requirements. It is likely that they will be included, but if necessary is is acceptable to have an implementation that do not satisfy those requirements.

  • Nice to Have (NTH) -- These requirements are those which may provide some level of benefit but are not critical to the implementation of this feature. They may be included in the implementation of this feature if time permits, but priority must be given to the "must have" and "highly desirable" requirements.


The words method and means identify some kind of technique by which the feature will be supported. They do not specifically imply either an API (programming language method or procedure) or a command.

1.5 Overview

The General Description section describes the general factors that affect the requirements which are defined in detail in the Specific Requirements section, and makes them easier to understand. The Specific Requirements contain the specific requirements for this release.

 

2. General Description

2.1 Feature Perspective

This document contains requirements and suggested functionality for the OpenDS client tools implementation. It describes the overall strategy for OpenDS client tools. In particular, this document addresses the following issues:

  • local access versus remote access

  • single server versus group of servers

  • single versus multiple command lines

  • single versus multiple server configuration and/or management

  • platform dependencies

2.2 User Characteristics

The following schema represents the overall administration and configuration framework architecture (extract from Administration and Configuration Framework Architecture document):

User who wants to perform a management operation against a given server instance, or a group of servers, will use a client tool. Administrator credentials, central repository location, server or group of server identification should be consistent across the different management dimension (local versus remote,  single server versus group of servers, ...) and the different client tools.

2.3 Constraints

In order to manage a server, a registration phase will be required. This registration procedure will be described in another document.

3. Specific Requirements

3.1 Administration Requirements

The client tools as a whole must satisfy certain requirements however, all client tools may not necessarily satisfy every requirement.
  1. (MH) OpenDS client tools must expose the same user interface on all platforms.

  2. (MH) OpenDS client tools must exhibit a uniform definition and consistent syntax (where applicable) and form.

  3. (MH) OpenDS client tools must work either if the server instance is local (on the same host) or remote.

  4. (MH) OpenDS administrative client tools must provide features for defining groups of administrative users.

  5. (MH) OpenDS administrative client tools must provide features for assigning roles and capabilities to a group of administrative users.

  6. (MH) OpenDS administrative client tools must provide features for assigning an administrative user to a group.

  7. (MH) OpenDS administrative client tool user authentication must be based on administrative users and/or groups of administrative users.

  8. (MH) OpenDS administrative client tools must provide a unified way to identify OpenDS server instance or groups of instances.

  9. (MH) OpenDS administrative client tools must operate, if applicable, on a single instance as well as on a group of instances.

  10.  (MH) OpenDS administrative client tools must support local or remote management of server(s) life cycle (start stop, ...).

  11. (MH) OpenDS administrative client tools must support local or remote modification of server(s) configuration.

  12. (MH) OpenDS client tools must include an administrative command line interface capable of satisfying all client tool requirements.

  13. (MH) OpenDS client tools must include a CLI for directory data queries

  14. (HD) The OpenDS administrative client tools should support local or remote configuration of server(s) if the server(s) is/are off-line or not started.

  15. (HD) OpenDS administrative client tools should include an HTML-based GUI.

  16. (HD) OpenDS HTML-based GUI should satisfy all client tool requirements.

  17. (HD) Client tools should provide a common configuration file for defaulting commonly specified options as required address/port of a server system and whether to use SSL or StartTLS

  18. (NTH) OpenDS administrative client tools may include a configuration shell.

  19. (NTH) OpenDS administrative client tools may include plug in modules for common integrated development environments (IDE).

  20. (NTH) OpenDS administrative client tools may support the ability to present directory data in graphical form via a data browser.

3.2 Performance Requirements

No specific requirements are defined.

3.3 Scalability Requirements

The tools must be capable of handling arbitrary number of remote server instances.

3.4 Internationalization Requirements

  1. (MH) Client tools should be fully internationalized.

3.5 Help Requirements

  1. (MH) Administrative client tools be provide on-line help

  2. (MH) Man pages should be provides for each administrative client tool

3.6 Security Requirements

  1. (MH) Administrative client tools be capable of encrypting all network traffic
